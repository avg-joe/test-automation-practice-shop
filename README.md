# Test Automation Practice Shop

A fully featured, intentionally testable e-commerce storefront built for **automation testing workshops**. Learners practice locating elements, handling async behaviour, debugging flaky tests, and detecting regressions — all without touching a real payment system.

## Project Purpose

This site is a **practice target** for automation testing workshops. Its goals:

- Teach locating elements via stable `data-testid` attributes
- Demonstrate handling network delays and async operations (via MSW)
- Let learners recognise and debug flaky tests (10 % random failure on `/api/checkout`)
- Provide a controllable regression scenario via the `PUBLIC_BREAKING_CHANGE` flag

Every interactive element exposes a `data-testid` attribute generated by `getTestId()`. When `PUBLIC_BREAKING_CHANGE=true` those IDs are scrambled or removed, simulating a real-world regression.

## Environment Flags

Create a `.env` file at the project root (or set variables in your Cloudflare Pages dashboard).

### `PUBLIC_BREAKING_CHANGE`

| Value | Behaviour |
|---|---|
| `false` (default) | All `data-testid` attributes return their stable IDs |
| `true` | IDs are scrambled or removed — locators break, teaching regression detection |

```bash
# .env
PUBLIC_BREAKING_CHANGE=true
```

> **Tip:** Toggle this flag between test runs to create a "before / after regression" exercise for your team.

## MSW API Contract

All API calls are intercepted in the browser by Mock Service Worker. No real server is involved.

### `POST /api/login`

| Field | Detail |
|---|---|
| Delay | 1 000 ms |
| Success | body `{ username: 'student' }` → `200 { success: true, token, user }` |
| Failure | any other username → `401 { success: false, message }` |

### `POST /api/cart/add`

| Field | Detail |
|---|---|
| Delay | 500 ms |
| Payload | `{ id, name, price, emoji }` |
| Response | `200 { success: true, cart, itemCount }` |

### `POST /api/cart/update`

| Field | Detail |
|---|---|
| Delay | 300 ms |
| Payload | `{ id, quantity }` |
| Response | `200 { success: true, cart, itemCount }` |

### `POST /api/cart/remove`

| Field | Detail |
|---|---|
| Delay | 300 ms |
| Payload | `{ id }` |
| Response | `200 { success: true, cart, itemCount }` |

### `POST /api/cart/clear`

| Field | Detail |
|---|---|
| Delay | 300 ms |
| Response | `200 { success: true, cart: [], itemCount: 0 }` |

### `POST /api/coupon/apply`

| Field | Detail |
|---|---|
| Delay | 500 ms |
| Payload | `{ code }` |
| Success | `200 { success: true, code, discountPercent, freeShipping }` |
| Failure | `400 { success: false, message: 'Invalid coupon code' }` |

### `POST /api/shipping`

| Field | Detail |
|---|---|
| Delay | 500 ms |
| Payload | `{ firstName, lastName, email, address, city, state, zip, country, method }` |
| Response | `200 { success: true, message, method }` |

### `POST /api/checkout`

| Field | Detail |
|---|---|
| Delay | 800 ms |
| Random failure | 10 % chance → `500 { success: false, message }` |
| Forced failure | Send header `X-Force-Failure: true` → always `500` |
| Success | `200 { success: true, orderId, message }`, clears in-memory cart |

### `POST /api/contact`

| Field | Detail |
|---|---|
| Delay | 600 ms |
| Payload | `{ firstName, lastName, email, phone, subject, message }` |
| Response | `200 { success: true, message, ticketId }` |

## Test Credentials & Coupons

### Login

| Field | Value |
|---|---|
| Username | `student` |
| Password | any value (not validated) |

Any other username returns `401 Invalid credentials`.

### Coupon Codes

| Code | Effect |
|---|---|
| `SAVE10` | 10 % off subtotal |
| `SUMMER20` | 20 % off subtotal |
| `FREESHIP` | Free shipping |

### Forcing a Checkout Failure

Add the request header `X-Force-Failure: true` to `POST /api/checkout` to guarantee a `500` response. Useful for writing deterministic error-state tests.

## Known Intentional Flakiness

These behaviours are **by design** — do not file them as bugs.

### 1. Random 10 % checkout failure

`POST /api/checkout` fails with `500` approximately 10 % of the time. This teaches automation engineers to:

- Write resilient tests with retry logic
- Distinguish genuine failures from intermittent ones
- Use `X-Force-Failure: true` header when a deterministic `500` is needed

### 2. Breaking-change mode (`PUBLIC_BREAKING_CHANGE=true`)

When this flag is `true`, `getTestId()` returns scrambled or missing IDs. Any test that hard-codes a `data-testid` locator will fail. This simulates a real-world regression where a developer accidentally removes or renames test hooks.

Use this flag to:

- Demonstrate the value of stable test-ID conventions
- Practice debugging broken locators
- Run regression exercises during workshops

```
public/
  mockServiceWorker.js   # MSW service worker (generated, do not edit)
  favicon.svg
docs/
  testids.md        # Exhaustive data-testid reference table
```

## Deploy (Cloudflare Pages)

The project is configured for Cloudflare Pages via `wrangler.toml`.

```bash
# Preview build locally
npm run build
npm run preview

# Deploy via Wrangler CLI
npx wrangler pages deploy dist
```

Set `PUBLIC_BREAKING_CHANGE` as an environment variable in the Cloudflare Pages dashboard under **Settings → Environment Variables** to toggle breaking-change mode per deployment environment.
